# Situational Center UX

Живой набор правил и готовых шаблонов, собранный из пофазной переделки Sigma Demo leader-view. Не абстрактная теория — только то, что уже прошло приёмку на реальном сценарии (термический / вода / СКУД / ЕДДС Кольцово).

## Психологическая модель оператора

Оператор за 3 секунды должен ответить себе на три вопроса — в этом порядке:

1. **Тревожиться или нет?** → отвечает Status Banner.
2. **Что делать прямо сейчас?** → отвечает чек-лист «Что делать / Чего не делать».
3. **Что будет, если не действовать?** → отвечают стрелки прогноза.

Всё остальное — детали, которые оператор читает **только после** ответа на эти три. Детали живут в `DetailReveal` / «Развернуть» модалках.

## F-паттерн и места, куда НЕ кладут работу

Левый верхний угол мозг читает как «хром браузера» / логотип. Туда **нельзя** класть:

- кнопки управления воспроизведением (шаг/сброс) — они «теряются»
- переход в каталог сценариев — это навигация, а не рабочий инструмент
- статус инцидента — его нужно видеть первым

Правильно: левый верх — либо логотип-watermark (14 px, нейтральный), либо пусто. Рабочая зона начинается ниже и правее.

## Status Banner (первое, что видит глаз)

Полная ширина, 20–30% высоты экрана, фоновая подсветка по `Criticality`:

| Criticality | Фон | Иконка-лампа | Заглавная надпись |
| :-- | :-- | :-- | :-- |
| normal | `bg-emerald-50/85` | CheckCircle / emerald-600 | СИСТЕМА В НОРМЕ |
| watch | `bg-amber-50/85` | Eye / amber-600 | НАБЛЮДЕНИЕ |
| elevated | `bg-orange-50/90` | Warning / orange-600 | ПОВЫШЕННАЯ ГОТОВНОСТЬ |
| high | `bg-red-50/90` | WarningOctagon / red-600 | ВЫСОКИЙ РИСК |
| critical | `bg-red-100/95` | Siren / red-700 | АКТИВНЫЙ ИНЦИДЕНТ |

Заголовок сценария (например «Опасность пожара в серверной») — **не главный заголовок**, а подзаголовок в правой части баннера.

## Цветовой код по семантике (не по данным)

Цвет несёт смысл действия, а не только тип данных. Операторский взгляд должен считывать функцию карточки за доли секунды:

| Семантика | Палитра | Пример |
| :-- | :-- | :-- |
| Информация / справка | sky (голубой) | i-кнопка, InfoModal, «Последнее событие» |
| Действия («делай») | emerald (салатовый) | «Что делать сейчас», прогноз с вмешательством |
| Запреты / риск | rose (розовый) | «Чего не делать», прогноз без вмешательства |
| Критичность | лестница emerald→amber→orange→red | лампы, полоски зон |
| Исполнение / службы | sky с акцентом | прогресс задач, список служб |

Иконки подкрашены в цвет своей семантики — не в нейтральный zinc. Так карточка читается даже без чтения заголовка.

## Зоны объекта — цветная полоса + мягкий тинт

Каждая зона — карточка с `border-l-4` в цвете своего состояния и пастельным фоном того же оттенка (`bg-emerald-50/55`, `bg-amber-50/70` и т. д.). Иконка зоны и статус-круг подкрашены в `criticalityText`.

Это решает проблему «три одинаковых строки с зелёной точкой — норма и тревога визуально неразличимы».

## Do / Don't чек-лист вместо абзаца

Секция «Рекомендуемые действия» разбивается на три блока с цветной семантикой:

1. **Что происходит** (карточка с полосой в цвет критичности, фон по критичности) — короткий тайтл + localizationStatus + чипы служб.
2. **Что делать сейчас** (emerald-тинт, CheckCircle-иконка) — до 3 нумерованных пунктов из `recommendations`, каждый с полем «Исполняет: …».
3. **Чего не делать** (rose-тинт, Prohibit-иконка) — 2 правила на `RiskKind` из статического словаря `doNotByRisk` (термический / вода / воздух / безопасность / операционный).
4. **Исполнение служб** (sky-тинт, ShieldCheck) — прогресс задач и ключевая задача.

Правила «чего не делать» пришли из реального регламента НГУ и операционной практики; хранятся в коде как `Record<RiskKind, string[]>`, а не в данных сценария.

## Прогноз — стрелки, не абзацы

Вместо двух параграфов «С вмешательством… / Без вмешательства…» — две карточки рядом:

- **С вмешательством** — TrendDown-иконка (emerald), подпись «Риск снижается», чип «Предотвращено: …»
- **Без вмешательства** — TrendUp-иконка (rose), подпись «Риск растёт», чип «Окно ухудшения: …»

Визуальная стрелка считывается мгновенно, текст — уже как уточнение.

## Sigma Assist — фиксированная панель справа-внизу

Весь «навигационный и вспомогательный слой» переносится в floating-панель в правом нижнем углу:

- Логотип (градиентный Robot-значок)
- Текущий сценарий + i-кнопка «Обстановка»
- Прогресс-бар «Этап X из Y»
- Транспорт: Шаг / Сброс
- 3 AI-подсказки с готовыми ответами из state (`explainability.causeEffectSummary`, `forecast.withoutSigma`, `forecast.withSigma`) — для демо-стенда без бэкенда
- Кнопка «Каталог сценариев»

Позиция `fixed; bottom: 16; right: 16; z-index: 30`, внутренний `max-h-[min(82vh,660px)] overflow-y-auto`. Сворачивается в pill-FAB по кнопке ∨. В `display` (видеостена, interactive=false) транспорт и каталог скрываются, AI-подсказки остаются — посетитель демо-стенда может ими пользоваться.

## Каталог сценариев — правый drawer, не центральная модалка

Drawer справа (`max-w-md`, `h-full`, sticky-хедер, 1 колонка карточек) стыкуется с Sigma Assist в одном углу экрана — оператор взаимодействует со всей навигацией одной рукой справа. Backdrop-клик закрывает, карточки в 1 колонку вместо 3.

## Чипы риска — по стихии

Чипы фильтра рисков окрашиваются по «стихии», не по нейтральному zinc:

| RiskKind | Цвет | Почему |
| :-- | :-- | :-- |
| thermal | orange | огонь |
| water | sky | вода |
| air | teal | бирюза воздуха |
| security | emerald | «зелёный свет» прохода охраны |
| operational | indigo | «внутренний» цвет, не стихия |

Активный чип — залитый цветом, неактивный — пастельный фон + цветная иконка + цветная рамка, пустой (0 сценариев под фильтр) — серый. То же семантическое правило применяется к левой полосе карточки сценария в результатах.

## Компактные кнопки действий

Первичное действие — pill-кнопка с иконкой и текстом («▶ Открыть», `px-2.5 py-1 text-[11px]`). Вторичное действие — **квадратная иконная кнопка** (`h-7 w-7`) с Television / Projector иконкой и `<span className="sr-only">…</span>` для a11y и автотестов. Это освобождает вертикальное пространство в карточках и снимает необходимость переноса строки на узком drawer.

## Информационная i-кнопка

- Сама кнопка — `bg-sky-50`, `text-sky-600`, круглая 24 px, с Info-иконкой
- Модалка — `bg-sky-50/95` фон, белая карточка с текстом внутри, иконка «i» в заголовке, источник — `sky-700`
- Закрывается по Escape, клику вне и ✕-кнопке
- Текст тела — `whitespace-pre-line` для многоабзацных регламентов

i-кнопки ставятся у заголовков секций «Обстановка» и «Рекомендуемые действия». Данные — из `scenarioReferences` в коде (отдельно `situation` и `actions`). Содержимое — реальные выдержки регламентов (Положение ЕДДС Кольцово, инструкция НГУ Приложение №6) с указанием номеров пунктов.

## Тренажёрный экран (Phase 4.d)

Отдельный маршрут `/trainer/:scenarioId` с собственной раскладкой — не переиспользует `LeaderDashboard` целиком, но **забирает словарь цветов** (`criticalityText`, `criticalityAccentBorder`, sky-info / emerald-do / rose-prohibited). Правила:

- **Верх экрана — шапка-карточка** с заголовком сценария + карточка счёта (sky-тинт) + кнопки «Заново» / «Выйти». Не повторяем Status Banner, чтобы не отвлекать — ставка на фокус оператора
- **Ядро — шаг решения** в `Surface` с `border-l-4` по критичности шага (`step.systemCriticality`). Блок `narrative` сверху, заголовок «Ваше решение» ниже, кнопки `allowedActions` — вертикальный стек с `Sparkle` иконкой в sky
- **Подсказка** — amber-тинт (`bg-amber-50 border-amber-200`), скрыта по умолчанию, раскрывается кнопкой «Подсказка». Тело — `rationale` ожидаемого действия + `clauseRef` шага. Amber нарочно — это не запрет и не действие, это «внимание-смягчение», отдельная семантика
- **Итоговый экран** — список шагов с `verdictCopy`: emerald ✓ «Корректно», sky ✨ «Нейтрально», rose ⛔ «Противорегламентно». Выражает scoring в цвете, не только в цифрах
- **Baseline-шаги без `interactiveMeta` пропускаются** через `findNextInteractiveStep` — оператор не ждёт «вводных кадров»

Тренажёр не требует синхронизации operator↔display и не использует `playbackStore` — держит session state локально в `useState`. Это снижает связность и позволит Phase 4.e хранить результаты как plain JSON для экспорта.

## Стратегия скролла для видеостены

Телевизоры на стене сейчас часто с тачем — можно закладывать тап как часть сценария, но основной вид **не должен требовать скролла страницы**. Решения:

- Внутренний скролл для длинных панелей (`max-h + overflow-y-auto` на Surface), а не скролл страницы
- Sticky-компоненты (ControlRail, заголовок drawer) с собственным overflow
- `line-clamp-3` на длинных текстах вроде `statusLine` зоны, `subtitle` сценария
- Вместо `DetailReveal` аккордеона — отдельная «Развернуть в модалку» кнопка на тач-дисплее: полный текст открывается в InfoModal-like оверлее, закрывается вне-кликом

## Применение на тестах

- Все критичные testId сохраняются при редизайне (`operator-shell`, `decision-summary`, `forecast-summary`, `scenario-launcher`, `open-launcher`, `venue-card-*`, `launcher-apply`).
- Текстовые селекторы вроде `getByText(/этап X из Y/i)` работают, когда прогресс-бар в SigmaAssist — но SigmaAssist должен быть **внутри** `operator-shell` (иначе `within(shell).getByText` не найдёт).
- Иконные кнопки используют `sr-only` для совместимости с `getByText`/`getByRole`.

## Что НЕ трогаем

- Доменный слой: `criticality` → действия / службы / эскалация — это территория `smart-city-analyst`
- Типы/инварианты сценариев (`InvariantProfile`, `Storyboard kinds`) — на них завязаны тесты консистентности
- Содержимое регламентов — изменяется только при обновлении источника (реальный документ НГУ / Кольцово)

## Короткий чек-лист при любой правке leader-view

1. Первое, что видит глаз — статус (цвет, размер 20–30% высоты).
2. Второе — что делать (чек-лист, не абзац).
3. Третье — что будет без действия (стрелка, не текст).
4. Левый верх — нет рабочих инструментов.
5. Цвет несёт смысл функции, не только данных.
6. Навигация — в правом углу, рядом с assist.
7. Любая плотная карточка — с полосой в цвет состояния слева.
8. Длинный текст → `line-clamp` + «Развернуть» в модалку.
9. Вторичная кнопка — иконная + `sr-only` текст.
10. Тесты зелёные: `npm run lint && npm run test:run && npm run build`.
